# Preview 模块产品方案

## 1. 模块定位

Preview 不是单纯的 PDF 阅读器，也不是做题讲解模块，而是一个面向学生课前学习的 AI 预习工作台。

它要解决的核心问题只有三个：

1. 这份材料主要讲什么。
2. 我应该先看哪些知识点。
3. 哪些地方最容易不懂，并且我可以继续追问什么。

因此，Preview 的主体验不应是“打开文件”，而应是“把材料转成可预习的知识结构和对话入口”。

## 2. 产品目标

### 2.1 用户目标

用户进入预习模块时，通常处于以下三类场景：

1. 没有现成材料，只想快速预习一个专题，例如分数运算、行程问题。
2. 手头有讲义、教材、练习册，希望上传后快速看懂重点。
3. 看到一道题、一页教材或一张板书，想拍照后立即得到知识点解释。

### 2.2 产品目标

第一版预习模块应做到：

1. 支持“快捷预习”“上传讲义”“拍照预习”三种入口。
2. 统一进入“预习结果页”，而不是直接停留在文件本身。
3. 支持知识点提取和围绕知识点的预习对话。
4. 支持从 OCR 结果进入预习分析。
5. 让前端始终有明确的下一步操作，不出现“上传后不知道该做什么”的空状态。

## 3. 设计原则

1. 预习优先，不把 Preview 做成通用阅读器。
2. 结果优先，上传后优先展示 AI 结构化结果，再决定是否查看原文。
3. 入口清晰，保证用户在 1 到 2 步内进入可交互状态。
4. 前后端解耦，文件解析、OCR、知识点提取、对话分层实现。
5. 第一版先保证链路闭环，再补阅读器增强能力和悬浮工具。

## 4. 前端产品方案

### 4.1 一级入口设计

预习首页建议固定为三个主入口加一个历史入口：

1. 快捷预习
2. 我的讲义
3. 拍照预习
4. 最近预习

推荐首页结构：

1. 顶部标题区：今日建议、最近专题或学习提醒。
2. 中部快捷入口区：专题卡片、上传按钮、拍照按钮。
3. 下部最近记录区：最近打开的讲义、最近对话、最近知识点。

### 4.2 快捷预习

快捷预习用于解决冷启动问题，也是最适合展示产品价值的入口。

建议预置一批专题卡，而不是先维护完整讲义库。第一版可先放：

1. 分数运算
2. 行程问题
3. 工程问题
4. 比和比例
5. 百分数应用
6. 一元一次方程

用户点击专题后，进入专题预习页，页面展示：

1. 主题简介
2. 先修知识
3. 核心知识点列表
4. 常见易错点
5. 典型例题
6. 开始提问按钮

第一版可以由前端内置这批专题内容，或由后端维护静态配置后返回；不要求一开始就做内容运营后台。

当前闭环已经可用，但展示形态仍然不是最终想要的“预习讲义”。

下一轮应将“快捷预习”明确改造成“文章式自学资料”体验：

1. 用户点击专题后先秒开本地讲义，不阻塞等待 AI。
2. 讲义页不是“学习目标”“核心讲义”“掌握路线”这类卡片堆叠，而是一篇适合学生连续阅读的结构化文章。
3. 文章结构更接近自学资料，而不是后台知识点列表的前端翻译。
4. AI 在后台补充摘要、知识点和可追问上下文，但这些内容不应抢占讲义主界面。
5. 返回首页再进入同一专题时，优先复用本地缓存，不重复等待 AI。

目标中的讲义结构可参考下面这种形式：

```text
行程问题讲义

第一章：理解行程问题的基本概念

💡 核心公式
路程 = 速度 × 时间

正文导入段落……

1. 什么是速度？
解释段落……

📝 想一想
一个带入公式的简短思考题……

2. 线段图画法
解释段落……

[讲义配图或示意图占位]
```

这意味着快捷预习的重点不再是“先给出几张知识点卡片”，而是：

1. 先给学生一篇能顺着读下去的讲义。
2. 在讲义内部自然嵌入公式、例子、思考题、配图占位和小标题。
3. 把 AI 作为讲义阅读过程中的辅助能力，而不是首屏主体。

### 4.3 我的讲义

该入口支持用户上传 PDF、Word、图片等材料。

上传后不建议直接打开阅读器，而应先进入“预习结果页”。推荐流程：

1. 用户上传文件。
2. 前端展示“解析中”状态。
3. 后端完成文本抽取和知识点提取。
4. 前端进入预习结果页。
5. 用户再决定是否查看原文。

预习结果页建议包含：

1. AI 摘要
2. 知识点列表
3. 重点难点
4. 建议预习顺序
5. 推荐追问问题
6. 查看原文按钮

### 4.4 拍照预习

拍照预习适用于教材页、练习页、板书、课堂截图等场景。

推荐流程：

1. 用户拍照或上传截图。
2. 前端将图片传给 shared OCR。
3. OCR 返回 question_text。
4. question_text 进入 Preview 的知识点提取。
5. 前端展示预习结果页。

这一条链路应作为第一版重点，因为它和当前 shared OCR 能力最匹配。

### 4.5 阅读态与提问方式

第一版不建议把“PDF 阅读器 + 悬浮球截屏”作为核心体验，但讲义页本身应当具备明确的阅读态设计。

原因：

1. 系统级悬浮窗权限打扰强。
2. 悬浮球交互复杂，开发和测试成本高。
3. 它更像增强能力，不是 MVP 必需能力。

更合适的路线是：

1. 预习讲义页进入后默认隐藏底部栏，让页面更像沉浸式阅读，而不是普通 tab 页。
2. 顶部栏常驻，至少包含返回按钮、当前讲义标题和必要的更多操作入口。
3. 文末固定提供一个明显的“进入 AI 对话”按钮，作为最稳定的提问入口。
4. 如果端上支持划选文本，则允许用户在讲义页直接选中文字，并出现“去问 AI”入口，体验上接近系统复制/全选菜单。
5. 如果系统级文字选择菜单定制成本过高，则先保底实现“文末按钮进入 AI 对话”，后续再补划选提问。
6. 第一版先不做系统级悬浮球。
7. 如果需要阅读器，只做“查看原文”的辅助能力。
8. 更复杂的跨应用截屏或系统级悬浮能力继续后置评估。

#### 4.5.1 划选文字去问 AI 补充方案

这项能力的目标不是替代文末按钮，而是在学生阅读讲义时，允许他对某一句话、某一段解释或某一个公式说明直接发问。

交互目标：

1. 用户在讲义正文中长按并拖动，像系统选择文字一样完成划选。
2. 划选完成后，除了系统已有的复制、全选能力外，还应出现“去问 AI”入口。
3. 用户点击“去问 AI”后，跳转到独立 AI 对话页，而不是在当前阅读页内弹出复杂对话框。
4. AI 对话页顶部或首条消息附近展示“当前提问片段”，让用户知道自己正在追问哪段内容。

第一阶段实现建议：

1. 先把讲义正文组织为可连续选择的文本块，并统一放在选择容器中，保证基础划选体验稳定。
2. 先保留文末“进入 AI 对话”按钮作为保底入口，避免划选方案尚未稳定时影响主流程。
3. 划选能力第一版不强求一定把“去问 AI”直接塞进系统原生菜单；如果 Compose 对原生选择菜单扩展成本过高，则先采用“划选后出现一个轻量浮层按钮”的方式实现。
4. 这个轻量按钮应尽量靠近选区或固定出现在屏幕下沿，文案明确为“去问 AI”，避免与复制操作混淆。
5. 用户点击后，前端把“专题标题 + 章节标题 + 选中文本 + 当前讲义上下文”一起带入 AI 对话页。

第二阶段增强建议：

1. 如果端上实现成本可控，再尝试把“去问 AI”并入接近系统复制/全选的上下文菜单体验。
2. 如果未来需要更高精度，可在对话请求结构中补充 `selected_excerpt`、`section_title` 之类的字段，而不只把选中文本拼进 `context_text`。
3. 如果同一讲义存在配图或公式块，也可以进一步支持“对图下说明文字发问”或“对公式说明发问”。

对当前代码结构的落地方式建议如下：

1. 讲义阅读页继续作为 Preview 模块内部页面，不额外新开全局模块。
2. 底部栏隐藏应由全局 `MainScaffold` 根据 Preview 内部阅读态/对话态统一控制，而不是在讲义页里用视觉覆盖硬挡住。
3. 讲义正文选择能力优先落在 Preview 自己的文章组件层，而不是散落在多个小卡片组件里。
4. AI 对话页继续复用现有 `/api/preview/chat`，第一版先走前端拼接上下文；只有当命中率不够时，再扩接口。

### 4.6 预习讲义页

这是 Preview 最核心的页面，快捷预习至少应当先落到这里。它不是“AI 结果页”，而是“可阅读的预习讲义页”。

页面建议由五个区域组成：

1. 顶部常驻栏：返回、讲义标题、必要操作。
2. 讲义头图或标题区：专题名、章节名、导读。
3. 文章正文区：公式高亮、解释段落、编号小节、思考题、配图占位。
4. 文末操作区：进入 AI 对话、继续阅读下一章、查看原文或重新生成。
5. 轻量 AI 状态区：只提示“AI 正在后台整理”或“AI 已准备好”，不抢正文位置。

讲义页应保证用户至少有三种明确操作：

1. 直接顺着文章阅读并理解专题内容。
2. 在文末点击按钮进入 AI 对话。
3. 对正文中的某一段或某一小节发起提问。

讲义页在视觉和交互上应尽量接近截图中的阅读体验，而不是卡片列表页。

### 4.7 AI 对话页

AI 对话页应与讲义页彻底拆开，并尽量保持简洁。

不应出现下面这些会分散注意力的模块：

1. “AI 重点整理”
2. “对话聚焦”
3. “从讲义发起”
4. 额外的知识点大卡片区

AI 对话页建议只保留：

1. 顶部常驻栏：返回讲义页、标题。
2. 对话消息流。
3. 输入框和发送按钮。
4. 如果用户是从划选文字进入，则在首条消息或输入区上方展示“当前提问片段”。

这页的目标是让学生专注于“问”和“理解”，而不是再次浏览一套讲义摘要。

## 5. 后端产品方案

### 5.1 当前后端职责划分

Preview 模块只负责文本理解，不负责 OCR 本身，也不负责做题引导和错题整理。

当前边界建议保持为：

1. shared 负责图片 OCR。
2. preview 负责文本知识点提取和预习对话。
3. guide 负责解题引导。
4. mistakes 负责错题沉淀。

### 5.2 当前已实现能力

当前代码中，Preview 后端已经具备以下能力：

1. `POST /api/preview/knowledge-points`
   - 输入教材文本、OCR 文本或手动输入文本
   - 输出知识点列表和摘要
2. `POST /api/preview/chat`
   - 输入用户问题、上下文和已选知识点
   - 输出面向预习场景的讲解和追问建议
3. `GET /api/preview/health`
   - 健康检查
4. 快捷预习知识点结果文件缓存
   - 使用专题标题、内容文本和模型名生成缓存键
   - 避免同一专题反复进入时重复调用 LLM
5. `qwen3.6-flash` 模型配置
   - 优先降低快捷预习首轮 AI 整理耗时

当前 shared 已具备以下能力：

1. `POST /api/shared/ocr`
   - 输入图片 base64
   - 输出 OCR 识别文本
2. `POST /api/shared/ocr/dispatch`
   - OCR 后根据目标模块转发
   - 当目标为 preview 时，可直接进入预习知识点分析

### 5.3 推荐后端链路

三种前端入口对应三条后端链路：

1. 快捷预习
   - 前端先展示内置的文章式专题讲义
   - 前端把专题种子内容传给 Preview，在后台请求 AI 摘要和知识点整理
   - Preview 优先命中文件缓存，未命中时再调用 `qwen3.6-flash`
   - 前端本地持久化 AI 结果，但讲义页首屏仍以文章内容为主
   - 当用户点击文末按钮或划选正文时，再进入独立 AI 对话页
2. 我的讲义
   - 文件解析层先抽文本
   - 前端或中间层把文本传给 Preview
   - Preview 返回预习结果
3. 拍照预习
   - 前端把图片传给 shared OCR
   - shared 返回 question_text 或直接 dispatch 到 preview
   - Preview 输出知识点和摘要

### 5.4 文件解析建议

Preview 模块本身不建议直接承担 PDF/Word 解析逻辑，但产品方案里必须预留这条链路。

建议的处理方式：

1. PDF/Word 解析作为单独的文件解析层，独立于 Preview。
2. 解析层输出统一文本结构，再喂给 Preview。
3. Preview 不关心文件来自 PDF、Word 还是 OCR，只接收标准化文本。

建议未来统一的文本输入结构至少包含：

1. `user_id`
2. `content_text`
3. `source_type`
4. `session_id`
5. `page_hint`
6. `document_id` 或 `material_id`

### 5.5 预习对话策略

预习对话的目标不是直接代替解题，而是帮助学生建立认知。

因此对话策略应强调：

1. 先解释概念，再给公式，再给简单例子。
2. 优先帮助理解“这是什么”“为什么这样学”。
3. 默认少做完整解题步骤，避免和 guide 模块职责重叠。
4. 如果用户来自讲义划选，应优先围绕选中文本解释，而不是重新概括整篇专题。
5. 输出 1 到 3 个适合继续追问的问题，降低用户表达成本。

### 5.6 日志和画像预留

第一版建议继续保留本地日志，作为后续画像或学习记录的基础。

建议记录：

1. 用户输入来源
2. 识别后的文本
3. 知识点结果
4. 用户提问
5. AI 回复
6. 时间戳

当前实现已经支持本地 JSONL 落盘，这一设计可以保留。

## 6. 前后端接口映射建议

### 6.1 第一版正式接口

Preview：

1. `GET /api/preview/health`
2. `POST /api/preview/knowledge-points`
3. `POST /api/preview/chat`

Shared：

1. `GET /api/shared/health`
2. `POST /api/shared/ocr`
3. `POST /api/shared/ocr/dispatch`

### 6.2 第一版前端调用建议

1. 快捷预习
   - 先打开本地文章式讲义页
   - 后台调 `POST /api/preview/knowledge-points` 预热 AI 结果
   - 用户点击文末按钮或划选提问时，再调 `POST /api/preview/chat`
2. 拍照预习
   - 优先调 `POST /api/shared/ocr/dispatch`
   - 或先调 `/api/shared/ocr` 再调 `/api/preview/knowledge-points`
3. 上传讲义
   - 文件解析完成后调 `POST /api/preview/knowledge-points`

## 7. MVP 范围

### 7.1 第一版必须完成

前端：

1. 预习首页三个主入口
2. 快捷预习专题卡片
3. 拍照预习入口
4. 文章式预习讲义页
5. 常驻顶部栏 + 隐藏底部栏的阅读态
6. 简洁 AI 对话页
7. 文末进入 AI 对话入口

后端：

1. 知识点提取接口
2. 预习对话接口
3. OCR 转预习的调用链
4. 本地日志记录

### 7.2 第一版可以暂缓

1. 完整 PDF 阅读器体验
2. 系统级悬浮球
3. 跨应用截屏提问
4. 复杂文档管理系统
5. 用户画像闭环

## 8. 第二版规划

第二版再考虑以下增强：

1. 讲义页内划选文字提问
2. 讲义页内局部引用和回跳定位
3. 多页内容摘要
4. 专题学习历史
5. 预习结果和 guide/mistakes 联动
6. Profile 模块接收预习行为数据

## 9. 当前推荐落地方向

如果按开发优先级排序，建议按下面顺序推进：

1. 先把“快捷预习”从卡片式结果页改造成文章式讲义页。
2. 再把独立 AI 对话页简化到只保留对话本身。
3. 然后补讲义页的划选提问能力。
4. 再把“拍照预习”接上 shared OCR。
5. 最后再补“上传讲义后生成预习结果页”。

一句话总结：

Preview 最合适的形态不是“AI PDF 阅读器”，而是“围绕专题、讲义和拍照材料的 AI 预习工作台”；上传和拍照只是输入方式，真正核心是知识点结构化、预习结果页和可继续追问的对话能力。

对当前快捷预习来说，还要再补一句：它的主界面首先应该是一篇“像自学资料一样能读下去的讲义”，而不是“几组标签、卡片和摘要块”。

## 10. 接下来每一步的执行规划

后续尽量按下面的规划推进 Preview 模块。前后端可以交替进行，也可以并行推进，但每一阶段都必须有明确的可验证产出，避免只完成代码而没有可演示结果。

### 10.1 执行原则

1. 后端优先保证接口稳定和可调试。
2. 前端优先保证页面结构和交互路径可见。
3. 每完成一阶段，都要至少完成一次联调或单端验收。
4. 不跳过验证直接做下一阶段，否则后面返工成本会明显升高。

### 10.2 关于“能不能实时看到效果”

后端：

1. 可以实时调试 FastAPI。
2. 推荐使用 `uvicorn app.main:app --reload` 在 `backend/` 目录下启动。
3. 启动后可以通过浏览器、Postman 或命令行直接调用接口，看返回值和日志落盘效果。
4. 因为当前 Preview 和 shared 已有基础接口，所以后端每加一个能力，都可以即时验证。

前端：

1. Android 项目可以构建，也可以查看效果，但“只构建”本身不能看到页面效果。
2. 想看真实页面，需要在 Android Studio 中运行到模拟器或真机。
3. 如果页面里写了 Compose Preview，也可以在 Android Studio 的预览面板查看局部组件效果。
4. 当前阶段即使后端还没完全接好，前端也可以先用静态数据或假数据查看页面结构和交互流程。
5. 等到联调阶段，再把静态数据逐步替换成真实接口调用。

结论：

1. 后端可以实时调试接口效果。
2. 前端可以实时查看 UI 效果，但前提是要运行到模拟器、真机或使用 Compose Preview，而不是只执行构建命令。

### 10.3 Phase 0：联调基线准备

目标：先保证前后端都有稳定的开发入口。

当前状态：

1. 前端已在 Android Studio 中完成安装运行，当前可展示静态预习页面。
2. 后端已在 `conda activate bluetutor` 环境下完成本地基线验证。
3. `GET /health`、`GET /api/preview/health`、`GET /api/shared/health` 已验证通过。
4. `POST /api/preview/knowledge-points` 与 `POST /api/preview/chat` 已验证可调用并返回 200。
5. `guide` 路由因缺少 `langchain_core` 依赖未注册，但不影响当前 Preview Phase 0 联调。

后端任务：

1. 确认 FastAPI 可以稳定启动。
2. 确认 `/health`、`/api/preview/health`、`/api/shared/health` 可访问。
3. 确认 `.env` 中模型配置可用。
4. 保持 `knowledge-points` 和 `chat` 接口可单独调用。

前端任务：

1. 确认 Android 项目能在 Android Studio 打开并运行。
2. 确认当前预习页能在模拟器或真机显示。
3. 先不接真实接口，保留当前静态页面用于占位验证。

验收方式：

1. 后端可以本地启动并返回健康检查结果。
2. 前端可以在设备或模拟器中看到当前预习界面。

### 10.4 Phase 1：快捷预习闭环

目标：先做出最容易演示、最不依赖上传和 OCR 的闭环。

当前状态：

1. 后端已支持前端预置专题通过 `source_type=quick_topic` 接入 Preview。
2. `PreviewKnowledgeRequest` 和 `PreviewChatRequest` 已支持 `topic_id`、`topic_title` 元数据。
3. Android 前端已实现“点击快捷预习专题卡片 -> 请求后端 -> 展示讲义页 -> 继续提问”的基础闭环。
4. 当前前端的专题页仍偏向块状信息组织，不是目标中的文章式自学资料页，这一部分需要继续重构。
5. Android 本地联调默认使用模拟器访问地址 `http://10.0.2.2:8000`；如果改用真机，需要替换为宿主机局域网地址。

后端任务：

1. 约定快捷预习的专题输入方式。
2. 如果专题内容暂时由前端内置，则后端继续复用现有 `POST /api/preview/knowledge-points`。
3. 如果需要后端维护专题数据，则新增一个轻量专题配置接口或静态配置文件。
4. 保证知识点提取结果稳定可展示。

前端任务：

1. 在预习首页实现“快捷预习”专题卡片区。
2. 点击专题后进入文章式预习讲义页，而不是卡片式结果页。
3. 讲义页要改成连续阅读结构，突出章节标题、核心公式、解释段落、思考题和配图占位。
4. 讲义页隐藏底部栏，顶部栏常驻。
5. 文末提供“进入 AI 对话”按钮，并预留正文划选提问入口。
6. AI 对话页改成极简形态，只保留顶部栏、消息流和输入区。
7. 接入 `knowledge-points` 和 `chat` 两个接口，完成问答闭环，但不让 AI 摘要喧宾夺主。

验收方式：

1. 用户点击“分数运算”或“行程问题”后，可以进入结果页。
2. 讲义页呈现为一篇结构化文章，而不是若干信息卡片。
3. 用户可以通过文末按钮进入 AI 对话。
4. AI 对话页足够简洁，不再重复展示摘要、聚焦标签和额外卡片。

执行建议：

1. 这是最推荐优先完成的一步。
2. 这一阶段完成后，Preview 就已经具备可展示的核心价值。

### 10.5 Phase 2：拍照预习接入 shared OCR

目标：打通“图片 -> OCR -> 预习结果”的链路。

后端任务：

1. 保持 `POST /api/shared/ocr` 和 `POST /api/shared/ocr/dispatch` 可用。
2. 确认 dispatch 到 preview 时返回结构稳定。
3. 如有必要，补充 OCR 失败、空文本、识别不清晰等错误提示字段。

前端任务：

1. 在预习首页增加“拍照预习”入口。
2. 实现图片选择或拍照后的上传流程。
3. 优先接 `POST /api/shared/ocr/dispatch`，减少前端中间拼接逻辑。
4. 成功后跳转到统一的预习结果页。

验收方式：

1. 用户上传一张教材页图片。
2. 前端能收到 OCR 后转发给 Preview 的结果。
3. 页面展示摘要和知识点，而不是只展示原始 OCR 文本。

执行建议：

1. 这一阶段完成后，Preview 和 shared 的联动价值才能真正体现出来。
2. 如果拍照能力暂时不做真机拍摄，也可以先支持从相册选图完成验证。

### 10.6 Phase 3：上传讲义并生成预习结果页

目标：把“我的讲义”从上传入口做成真正可用的预习入口。

后端任务：

1. 明确 PDF/Word/图片的解析边界。
2. 新增文件解析层，或先支持一种最容易实现的材料类型。
3. 将解析后的文本统一送入 Preview。
4. 如果内容较长，补充分段、截断或分页输入策略。

前端任务：

1. 实现“我的讲义”上传页面或入口。
2. 增加“解析中”状态。
3. 上传成功后跳转到预习结果页。
4. 预留“查看原文”入口，但这一阶段不强求做复杂阅读器。

验收方式：

1. 用户上传一份讲义。
2. 系统能在合理时间内返回摘要和知识点结果。
3. 用户可以继续提问，不需要重新输入整段材料。

执行建议：

1. 这一阶段可以先只支持 PDF 或图片中的一种，不需要一开始同时把 PDF、Word、图片全部做完。
2. 先跑通流程，再逐步扩展文件类型。

### 10.7 Phase 4：统一预习结果页和对话体验

目标：让三个入口最终汇聚成统一体验，而不是三套各自独立的页面。

后端任务：

1. 统一预习结果页所需的字段结构。
2. 补充材料来源、页码、会话标识等元数据。
3. 评估是否需要新增“重新生成摘要”或“获取推荐追问”接口。

前端任务：

1. 把快捷预习、拍照预习、我的讲义统一接入同一套结果页组件。
2. 统一对话区样式和状态管理。
3. 增加空状态、加载状态和失败状态。

验收方式：

1. 三种入口最终都进入同一类预习结果页。
2. 结果页字段统一，不因为来源不同而切换三套布局。

### 10.8 Phase 5：阅读器增强能力

目标：在主链路稳定后，再做增强，而不是提前陷入重交互开发。

后端任务：

1. 支持页码、选区文本、局部上下文等附加输入。
2. 视情况支持“本页重点提取”之类的补充接口。

前端任务：

1. 增加“查看原文”能力。
2. 如果做阅读器，只先做最基础的原文查看和页码定位。
3. 第二阶段再考虑选中文本提问、框选区域提问。
4. 系统级悬浮球放到更后面评估，不进入当前优先级。

验收方式：

1. 用户可以在预习结果和原文之间切换。
2. 用户能从原文中定位到对应内容。

## 11. 推荐的开发节奏

推荐按下面节奏推进，而不是前后端完全割裂：

1. 后端先提供稳定接口。
2. 前端先用假数据把页面结构跑通。
3. 页面结构稳定后，再把关键页面切到真实接口。
4. 每完成一个入口，就立即做一次联调。

更具体地说：

1. 快捷预习适合前后端同步推进。
2. 拍照预习适合后端先稳住 OCR 链路，再让前端接入。
3. 我的讲义适合后端先明确文件解析方案，再让前端做上传流程。

## 12. 后续执行约定

后续完善 Preview 模块时，尽量遵守下面的约定：

1. 优先完成文档中排在前面的阶段，不跨阶段做重功能。
2. 每次只推进一个主闭环，做完就验证。
3. 如果某一步需要调整方案，优先更新本 README，再继续开发。
4. 默认以后续工作都以这份 README 作为产品和开发路线参考。
